Repository & Branching

Mono-Repo Decision

  • Atomic cross-service changes — one commit, one PR
  • Shared infra code (Terraform, Helm) alongside services
  • Path-filtered CI isolates per-service pipelines
  • Single clone for full project visibility

Trade-off: coarser access control vs. simpler coordination

GitHub Flow + develop

feature/* ──► develop ──► main ──► Production hotfix/* ──► main (direct)

  • main — always deployable, protected
  • develop — integration branch
  • PRs require: CI pass + Trivy scan + 1 review

CI Pipeline Design

Per-service, path-filtered, fully independent

Frontend CI

Checkout→ npm install→ Lint→ Jest Tests→ Docker Build→ Trivy Scan→ ACR Push

Triggers: services/frontend/** on main/develop

Backend CI

Checkout→ Maven Build→ Unit Tests→ Docker Build→ Trivy Scan→ ACR Push

Triggers: services/backend/** on main/develop


Key: Image tags use Git SHA prefixes — every image maps to an exact commit. Images only pushed on main.

CD Pipeline — Deployment

Triggered on CI completion (main) or manually via workflow_dispatch

Azure Login→ Get AKS Creds→ Detect Inactive Slot→ Helm Deploy (MongoDB)→ Helm Deploy (BE)→ Helm Deploy (FE)→ Health Check→ Switch Traffic

CI/CD Platform: GitHub Actions

  • Zero setup — built into GitHub
  • First-party Azure actions
  • YAML declarative config
  • 2,000 min/month free tier

Why not Jenkins / GitLab CI?

  • Jenkins: self-hosted infra overhead, plugin management
  • GitLab CI: would require moving the repo
  • Azure DevOps: strong Azure integration, but adds org complexity

Performance & Scaling

Performance Measures

  • Multi-stage Docker builds — FE ~80 MB, BE ~150 MB
  • Readiness probes — traffic only to healthy pods
  • Connection pooling — MongoDB driver + HTTP keep-alive
  • Prometheus metrics — request latency histograms

Critique: Spring Boot cold start (15–30s) is mitigated by readiness probes and min 2 replicas. GraalVM native image would reduce to <1s but adds build complexity.

Security — Defence in Depth

Network

  • VNet isolation (dedicated subnet)
  • Calico network policies
  • No public IP on database

Container

  • Non-root execution
  • Minimal Alpine base (~5 MB)
  • Multi-stage builds (no source in prod)

Application

  • Helmet.js security headers
  • Jakarta Bean Validation
  • Rate limiting (100 req/15 min)

Supply Chain

  • Trivy scans every image in CI
  • Builds FAIL on CRITICAL CVEs
  • Managed Identity auth (no passwords)
  • ACR admin account disabled

Production Recommendations

  • Azure Key Vault for secrets
  • Pod Security Standards (Restricted)
  • Application Gateway WAF
  • cert-manager + Let's Encrypt TLS
  • Image signing (Notary v2)

Backup, Restore & Rollback

Backup Strategy

RPO: 24 hours  |  RTO: 1 hour
  • Kubernetes CronJob — daily at 02:00 UTC
  • mongodump → tar/gzip compressed
  • 7-day rolling retention
  • Target: Azure Blob Storage (LRS)

Critique: Add backup-to-Blob automation, monthly restore tests, encryption at rest. Consider Cosmos DB for built-in PITR.

Rollback Tiers

Tier 1Slot switch<10s
Tier 2Helm rollback<60s
Tier 3Previous image tag<5 min
Tier 4Full infra rebuild15–30 min

Tier 1 is instant — previous pods stay running. Switching the Service selector is a metadata-only operation.

Sustainability Measures

  • Burstable VMs — lower baseline energy
  • Autoscaling — pods + nodes scale down when idle
  • Scheduled teardown — destroy non-prod overnight (~65% compute savings)
  • Minimal images — less storage & transfer
  • Azure Emissions Dashboard — track carbon footprint per subscription

Most impactful: Automating overnight environment teardown via scheduled GitHub Actions.

Additional Features

1. Automated MongoDB Backup

Kubernetes CronJob runs mongodump daily with 7-day retention and auto-cleanup.

2. Horizontal Pod Autoscaling

HPA v2 with multi-metric scaling (CPU + memory) on frontend and backend services.

3. Container Vulnerability Scanning

Trivy integrated into CI — scans OS packages + language dependencies. Fails on CRITICAL CVEs.

4. Blue/Green Zero-Downtime Deploy

Pod label slot switching with automatic inactive-slot detection and instant rollback capability.

Critical Reflection

Strengths

  • Full automation — zero manual steps from commit to production
  • Independent pipelines — per-service CI with path filters
  • Instant rollback — blue/green slot switch in <10s
  • Security by default — non-root, minimal images, Trivy, managed identity
  • Observability — Prometheus metrics + centralised logging

Limitations & Future Work

  • Single MongoDB — needs Replica Set / Cosmos DB
  • No integration tests — add FE→BE→DB test stage
  • No service mesh — needed for canary, mTLS, tracing
  • No GitOps — ArgoCD/Flux for declarative cluster state
  • Manual rollback — automate with post-deploy health gate
  • No auth — add Azure AD / Entra ID for production
